ТЕХНИЧЕСКОЕ ЗАДАНИЕ
на разработку дипломного проекта
1. Наименование проекта
Проектирование и разработка веб-платформы онлайн-аукциона автомобилей с применением Clean Architecture и принципов SOLID
2. Актуальность темы
Рынок онлайн-продажи автомобилей характеризуется недостаточной прозрачностью торгов, отсутствием механизма защиты участников и высоким риском недобросовестных сделок.
Создание информационной системы, обеспечивающей:
•	конкурентную обработку ставок,
•	защиту участников через депозитный механизм,
•	прозрачность торгов,
представляет практический и исследовательский интерес.
3. Цель работы
Целью дипломной работы является разработка архитектурно корректной и масштабируемой веб-платформы онлайн-аукциона автомобилей с применением современных архитектурных подходов и принципов проектирования.
4. Задачи работы
В рамках дипломного проекта необходимо:
1.	Провести анализ предметной области онлайн-аукционов.
2.	Исследовать архитектурные стили проектирования backend-систем.
3.	Обосновать выбор Clean Architecture и Vertical Slice Architecture.
4.	Реализовать серверную часть системы на Java.
5.	Спроектировать модульную архитектуру с изоляцией бизнес-логики.
6.	Реализовать механизм конкурентной обработки ставок.
7.	Реализовать систему депозитов.
8.	Обеспечить транзакционную целостность данных.
9.	Провести тестирование системы.
10.	Выполнить анализ производительности и устойчивости.
5. Архитектурные решения
5.1 Общий архитектурный стиль
В проекте применяется:
•	Clean Architecture
•	Vertical Slice Architecture
•	Modular Monolith (единый deployable модуль с логической сегментацией)
5.2 Обоснование выбора архитектуры
Clean Architecture
Позволяет:
•	изолировать бизнес-логику от инфраструктурных деталей;
•	обеспечить независимость доменной модели;
•	повысить тестируемость;
•	снизить связанность компонентов.
Vertical Slice Architecture
Система структурируется по функциональным модулям (feature-based), а не по техническим слоям.
Каждый функциональный модуль содержит:
•	domain
•	application (use cases)
•	infrastructure
•	web
Это обеспечивает:
•	высокую когезию
•	слабую связанность
•	упрощение масштабирования

6. Структура системы
Проект состоит из следующих функциональных модулей:
•	user
•	auction
•	vehicle
•	deposit
•	diagnostic
•	admin
Каждый модуль реализуется как вертикальный срез.

7. Применение принципов SOLID
В проекте реализуются принципы:
S — Single Responsibility
Каждый класс выполняет одну логическую функцию.
O — Open/Closed
Расширение функциональности осуществляется через абстракции без изменения существующего кода.
L — Liskov Substitution
Наследуемые типы корректно заменяют базовые.
I — Interface Segregation
Используются узкоспециализированные интерфейсы.
D — Dependency Inversion
Зависимости направлены на абстракции (ports), а не на реализации.

•	Spring Boot 3
•	Spring Security (JWT)

•	WebSocket (STOMP)

•	Clean Architecture

•	Modular Monolith

9. Функциональные требования
9.1 Управление пользователями
Система должна обеспечивать:
•	регистрацию;
•	аутентификацию через JWT;
•	разграничение ролей;
•	хранение профилей пользователей.

9.2 Управление автомобилями
Система должна обеспечивать:
•	создание карточки автомобиля;
•	загрузку изображений;
•	указание характеристик;
•	прикрепление диагностического отчета.

9.3 Модуль аукциона
Система должна обеспечивать:
•	создание аукциона;
•	установку стартовой цены и шага;
•	обработку ставок;
•	продление времени аукциона при поздних ставках;
•	автоматическое завершение аукциона;
•	определение победителя.

9.4 Механизм конкурентной обработки ставок
Для обеспечения целостности данных используется:
•	транзакционная обработка;
•	optimistic locking;
•	контроль версий агрегатов.

9.5 Система депозитов
Система должна обеспечивать:
•	внесение депозита;
•	блокировку средств на период торгов;
•	удержание при нарушении обязательств;
•	возврат проигравшим участникам.

10. Нефункциональные требования
Система должна обеспечивать:
•	поддержку не менее 100 одновременных пользователей;
•	корректную обработку конкурентных запросов;
•	логирование операций;
•	защиту персональных данных;
•	отказоустойчивость на уровне транзакций.

11. Научная составляющая
В рамках работы исследуются:
•	методы обеспечения целостности данных при конкурентных операциях;
•	применение принципов SOLID в модульной архитектуре;
•	сравнительный анализ архитектурных подходов;
•	влияние архитектурного выбора на масштабируемость системы.

12. Ожидаемый результат
Результатом выполнения дипломной работы является разработанная и протестированная веб-платформа онлайн-аукциона автомобилей с архитектурой, обеспечивающей:
•	масштабируемость;
•	расширяемость;
•	устойчивость к конкурентным запросам;
•	изоляцию бизнес-логики от инфраструктуры.
